## 1. Migrate `#[allow(...)]` to `#[expect(..., reason = "...")]`

### Why

`#[allow]` silently keeps suppressing a warning even when the underlying code
no longer triggers it. `#[expect]` was introduced in Rust 1.81 specifically
to catch this — it warns when the expected lint did not fire, which means
stale overrides surface during code review instead of accumulating forever.

Today the workspace has roughly:

- `rustconn/src/`: ~310 `#[allow(...)]`, fewer than 5 `#[expect]`
- `rustconn-core/src/`: ~50 `#[allow(...)]`, 0 `#[expect]`
- `rustconn-cli/src/`: ~20 `#[allow(...)]`, 0 `#[expect]`

### Plan

1. Add the project-wide warning lint to `Cargo.toml`:

   ```toml
   [workspace.lints.clippy]
   # ...existing entries...
   allow_attributes_without_reason = "warn"
   ```

   This forces every override to carry a `reason = "..."` string, which is
   needed for the `#[expect]` migration anyway.

2. Run `cargo clippy --all-targets` to get the full list of unannotated
   overrides (will be many — that is expected).

3. Migrate file by file. Pattern:

   Before:
   ```rust
   #[allow(clippy::too_many_arguments)]
   pub fn build_command(...) { ... }
   ```

   After:
   ```rust
   #[expect(
       clippy::too_many_arguments,
       reason = "Protocol command builders mirror server-side layout; \
                 splitting into a struct hurts readability"
   )]
   pub fn build_command(...) { ... }
   ```

4. When the lint *should not* fire any more, remove the override entirely
   instead of converting it.

5. Special cases that must stay as `#[allow]`:
   - Inside macros that expand into user code (`#[allow]` survives macro
     hygiene; `#[expect]` does not).
   - On generated code — `#[allow]` on the `mod` line.

### Acceptance criteria

- `cargo clippy --all-targets` clean.
- Zero `#[allow(...)]` without a `reason` outside the two cases above.
- At least 80 % of the existing `#[allow]` migrated to `#[expect]`.

### Suggested PR shape

One PR per crate. Each PR is mostly mechanical and should be reviewed for the
quality of the `reason` strings, not for code logic.

## 2. RDP RemoteApp: stop passing `/p:` on the cmdline

### Why

Today `rustconn/src/embedded_rdp/launcher.rs` builds the FreeRDP 3.x command
like this:

```rust
if is_remote_app {
    cmd.arg(format!("/p:{}", password.expose_secret()));
}
```

This is the only path in the codebase where a password ends up in
`/proc/PID/cmdline`, visible to any other process of the same uid. We
documented it as a Known Issue in 0.14.10 — 0.15.0 should actually fix it.

### Constraints

- FreeRDP 3.x `/from-stdin` does work for desktop RDP but [breaks for
  RemoteApp / RAIL](https://github.com/FreeRDP/FreeRDP/issues/12485). That is
  why the cmdline path was kept.
- We cannot use environment variables — `xfreerdp` does not consume
  `RDP_PASSWORD` style env vars.

### Options to evaluate, in order of preference

1. **Smartcard / Kerberos passthrough.** If the user has an existing Kerberos
   ticket (`klist` non-empty) skip the password entirely and let GSSAPI do
   the work. Falls back to option 2 / 3 when no ticket is present.

2. **`freerdp-shadow-cli --password-stdin` style helper script.** Spawn a
   tiny shell helper that reads stdin and writes a temporary
   `~/.config/freerdp/{uuid}.rdp` file with the password, then `xfreerdp3 ~/.config/freerdp/{uuid}.rdp /app:...`.
   The temp file lives in `XDG_RUNTIME_DIR` (mode 0600) and is removed in a
   `Drop` impl on the launcher struct. RAIL-compatible because the cmdline
   no longer contains the secret.

3. **Make RemoteApp opt-in with an explicit warning.** If neither option 1
   nor option 2 is feasible, add a setting "Allow RemoteApp passwords on
   cmdline (insecure)" defaulted to off. When off, RemoteApp connections
   prompt the user to authenticate inside the published app instead.

### Plan

1. Spike option 2 in a throwaway branch — confirm `xfreerdp3` reads `.rdp`
   files reliably and that RAIL still works.
2. Implement the temp-file flow in
   `rustconn/src/embedded_rdp/launcher.rs`. New helper:

   ```rust
   /// Writes a single-use FreeRDP connection file with mode 0600 in
   /// `$XDG_RUNTIME_DIR`. The returned guard removes the file on drop
   /// even if the launcher panics or the process is killed (best-effort).
   pub(super) struct EphemeralRdpFile {
       path: PathBuf,
   }

   impl EphemeralRdpFile {
       pub(super) fn write(
           connection: &Connection,
           password: &SecretString,
       ) -> std::io::Result<Self> { ... }
   }

   impl Drop for EphemeralRdpFile { ... }
   ```

3. Update `launcher.rs` so the RemoteApp branch builds the file via
   `EphemeralRdpFile::write` and runs `xfreerdp3 /path/to/file.rdp
   /app:program:...`.
4. Add a property test that `EphemeralRdpFile::Drop` removes the file even
   when the inner string contains weird characters.
5. Remove the Known Issue from `CHANGELOG.md` for 0.15.0.

### Acceptance criteria

- `grep -R '/p:'` in `embedded_rdp/` returns no matches outside test code.
- Manual smoke: open a RemoteApp connection on Wayland and X11 — both work
  without any password on the command line.
- New property test passes under address sanitiser builds.

## 3. CLI command doc comments: add `# Errors`

### Why

Roughly 25 public functions in `rustconn-cli/src/commands/` return
`Result<(), CliError>` without documenting which error variants the caller
should expect. The Microsoft Rust guidelines (M-CANONICAL-DOCS) require an
`# Errors` section on every public `Result`-returning function.

### Files

```
rustconn-cli/src/commands/connect.rs        cmd_connect
rustconn-cli/src/commands/wol.rs            cmd_wol
rustconn-cli/src/commands/move_cmd.rs       cmd_move
rustconn-cli/src/commands/pin.rs            cmd_pin, cmd_unpin
rustconn-cli/src/commands/test.rs           cmd_test
rustconn-cli/src/commands/delete.rs         cmd_delete
rustconn-cli/src/commands/sync.rs           cmd_sync
rustconn-cli/src/commands/duplicate.rs      cmd_duplicate
rustconn-cli/src/commands/export_import.rs  cmd_export, cmd_import
rustconn-cli/src/commands/show.rs           cmd_show
rustconn-cli/src/commands/list.rs           cmd_list (formats)
rustconn-cli/src/commands/secret.rs         cmd_secret_get/set/delete
rustconn-cli/src/commands/sftp.rs           cmd_sftp
rustconn-cli/src/commands/group.rs          cmd_group_*
rustconn-cli/src/commands/add.rs            cmd_add
rustconn-cli/src/commands/update.rs         cmd_update
```

### Pattern to apply

Before:
```rust
/// Connects to a saved connection by name or UUID.
pub fn cmd_connect(args: ConnectArgs) -> CliResult<()> { ... }
```

After:
```rust
/// Connects to a saved connection by name or UUID.
///
/// # Errors
///
/// Returns:
/// - [`CliError::ConnectionNotFound`] when no connection matches the name or UUID
/// - [`CliError::Vault`] when the configured secret backend is unreachable
/// - [`CliError::Spawn`] when the protocol-specific client (ssh, xfreerdp,
///   vncviewer, …) cannot be launched
pub fn cmd_connect(args: ConnectArgs) -> CliResult<()> { ... }
```

Keep the descriptions short — list the variants and a one-line trigger,
nothing more.

### Acceptance criteria

- `cargo clippy --all-targets -- -D clippy::missing_errors_doc` passes for
  the `rustconn-cli` package.
- All public `cmd_*` functions list at least one error condition.

## 4. Manual `Debug` impls for secret backends

### Why

Every backend in `rustconn-core/src/secret/{libsecret,passbolt,pass,keepassxc,bitwarden,onepassword,macos_keychain}.rs`
plus `SecretManager` and `CredentialResolver` is a public type without
`Debug`. Using them in `tracing::debug!(..., backend = ?backend, ...)` falls
back to memory addresses, which is unhelpful when troubleshooting.

`#[derive(Debug)]` does not work because every backend holds either an
async-trait object, a runtime handle, or fields the user might not want
echoed in logs (`service_account_token`, `application_id`).

### Pattern to apply

```rust
impl std::fmt::Debug for BitwardenBackend {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        f.debug_struct("BitwardenBackend")
            .field("server_address", &self.server_address)
            .field("session_present", &self.session.is_some())
            // Never log the actual session key.
            .finish_non_exhaustive()
    }
}
```

For each backend file:

1. Add the manual `Debug` impl right after the struct definition.
2. Add a unit test in the same file:

   ```rust
   #[test]
   fn debug_does_not_leak_secret() {
       let creds = build_test_creds_with_password("hunter2");
       let backend = BitwardenBackend::with_credentials(creds);
       let rendered = format!("{backend:?}");
       assert!(
           !rendered.contains("hunter2"),
           "Debug output leaked the password: {rendered}"
       );
   }
   ```

### Acceptance criteria

- All seven backend types implement `Debug`.
- `SecretManager` implements `Debug` (just shows backend count and cache TTL).
- Each backend file has a `debug_does_not_leak_secret` test.

## 5. GUI spacing audit

### Why

GNOME HIG recommends spacing in 6 / 12 / 18 / 24 px steps. The codebase
currently has roughly 40 occurrences of `set_margin_top(4)` /
`set_margin_top(8)` / `set_spacing(4)` in `rustconn/src/window/` and
`rustconn/src/dialogs/`. This is mostly accidental — these blocks were
written before the project standardised on the HIG values.

### Files most affected

```
rustconn/src/window/templates.rs
rustconn/src/window/sessions.rs
rustconn/src/window/snippets.rs
rustconn/src/window/edit_group.rs
rustconn/src/dialogs/recording.rs
rustconn/src/dialogs/log_viewer.rs
rustconn/src/dialogs/snippet.rs
rustconn/src/dialogs/template_manager.rs
```

### Plan

1. Run a script that finds non-HIG spacing values:

   ```bash
   rg -n 'set_(margin|spacing)\w*\((\d+)\)' rustconn/src \
     | rg -v '\((6|12|18|24)\)'
   ```

2. For each match, decide:
   - "between related widgets" → 6
   - "default container padding" → 12
   - "between groups" → 18
   - "before/after large sections" → 24

3. When converting a value would visibly change layout (rare for ±2 px), add
   a one-line comment explaining the deviation, otherwise just change the
   number.

### Acceptance criteria

- The grep above returns zero results that are not visibly justified by a
  comment.
- Manual visual smoke test on a 1920×1080 screen and a 1366×768 screen.

## 6. `EntryRowBuilder::new()` i18n marker

### Why

Several connection dialogs use `EntryRowBuilder::new("Kubeconfig")` (and
similar). Inside `.build()` the string is wrapped in `i18n()`, so the call
runs correctly at runtime, but `xgettext` only sees the raw literal — and
because `i18n()` is not at the call site, those strings never make it into
`po/rustconn.pot`. As a result they remain untranslated forever in
`uk.po`, `de.po`, etc.

### Files with the issue

- `rustconn/src/dialogs/connection/kubernetes.rs`
- `rustconn/src/dialogs/connection/serial.rs`
- `rustconn/src/dialogs/connection/ssh.rs` (a few rows)
- `rustconn/src/dialogs/connection/telnet.rs`
- `rustconn/src/dialogs/cluster.rs`
- `rustconn/src/dialogs/smart_folder.rs`

### Plan

Two equally-valid fixes:

#### Option A — change the builder API (preferred)

Make `EntryRowBuilder::new` accept `&str` but require the **caller** to
pass a value already wrapped in `i18n()`:

```rust
// Before
EntryRowBuilder::new("Kubeconfig").build()

// After
EntryRowBuilder::new(&i18n("Kubeconfig")).build()
```

Inside the builder, simply pass the value through to `set_title` without
re-wrapping. This keeps the `i18n()` call at every call site, so xgettext
catches them all. The cost is a small mechanical rewrite of every call.

#### Option B — add a marker macro

Keep the current call sites but add a `_i18n_marker!()` block in
`rustconn/src/i18n_markers.rs` that lists every literal currently passed to
`EntryRowBuilder::new` — same trick already used elsewhere in the file.

This is one-line-per-string and avoids any API change, but is easy to forget
when adding new rows.

### Acceptance criteria

- `bash po/update-pot.sh` produces a POT containing every UI label in the
  affected files.
- `msgmerge --update po/uk.po po/rustconn.pot` shows new entries for the
  previously-missing strings (or none if they were already translated by
  hand).

## 7. Cleanup of `let _ = ...` without justification

### Why

Several places call into the connection or settings layer and discard the
`Result` with `let _ =`, with no comment explaining why. Examples:

- `rustconn/src/window/connection_actions.rs:283,291,300,333` — silently
  drops `delete_connection` / `update_group` errors.
- `rustconn/src/window/smart_folders.rs:51,105,142,169` — silently drops
  `update_settings` errors, so a settings save can fail without the user
  noticing.
- `rustconn/src/i18n.rs:354-356` — three `let _ = bindtextdomain(...)` in a
  row.

### Pattern to apply

For each occurrence, choose one of:

1. The error really is best-effort and unrecoverable → keep `let _ = …` but
   add a `// best-effort: …` comment on the same line.
2. The error is recoverable → replace with:

   ```rust
   if let Err(e) = state_mut.update_settings(...) {
       tracing::warn!(error = %e, "failed to update settings");
   }
   ```

   For settings saves specifically, also surface a toast on the active
   window so the user knows the change did not stick.

### Acceptance criteria

- `rg 'let _ = ' rustconn/src` returns only commented `let _ = …` lines.
- Settings-save failure shows a toast.

## 8. (Optional) New `Manager` types are renamed

### Why

The Microsoft Rust guidelines recommend avoiding `Manager` / `Service` /
`Factory` suffixes — they are inherited from OO languages and rarely add
information. Existing names like `ConnectionManager`, `SessionManager`,
`SecretManager` we keep for backwards compatibility. *New* types added in
0.14.x should not pick up the suffix.

### Candidates

| Current | Suggested |
|---------|-----------|
| `ClusterManager` | `ClusterRegistry` |
| `SmartFolderManager` | `SmartFolderRegistry` |
| `TabGroupManager` | `TabGroupAllocator` |
| `RecordingManager` | `RecordingLibrary` |
| `TunnelManager` | `TunnelRunner` |
| `SshAgentManager` | `SshAgent` (no other `SshAgent` trait exists) |
| `CredentialVerificationManager` | `VerificationLedger` |

### Plan

Use the IDE's rename refactor (semantic, not text). Each rename produces a
PR of its own; the diff should consist entirely of renames plus a `#[deprecated]` re-export of the old name when the type is in `pub use` of
`lib.rs`:

```rust
#[deprecated(since = "0.15.0", note = "renamed to TunnelRunner")]
pub type TunnelManager = TunnelRunner;
```

Keep the alias for one minor release, then drop in 0.16.0.

### Acceptance criteria

- New names compile, all callers updated.
- `#[deprecated]` aliases re-exported under the old names so external users
  do not break.
